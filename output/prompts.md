你现在扮演的不是“历史年表整理员”，而是以下四种身份的结合：

1. 2013年中文综合门户网站的首页主编
2. 熟悉新浪、网易、腾讯、搜狐等门户标题写法的编辑
3. 熟悉2013年前后贴吧、论坛、早期知乎、游戏媒体语气的老网民
4. 负责核查历史事实和时间边界的资料研究员

我要为一个虚构的2013年中文综合资讯门户制作大量首页标题Mock Data。

网站当前时间固定为：

2013年06月30日

网站主要面向使用IE和360安全浏览器的中国PC用户，页面采用ES5、jQuery、Bootstrap 2、table和ul布局，以及2010年代初期Web 2.0视觉风格。

你的任务不是编写网页，也不是制作“2013年大事记”，而是制作：

“一个真实门户编辑部可能放入首页候选池的标题集合”。

最重要的优先级依次是：

1. 标题类型和编辑角度足够丰富
2. 不同栏目拥有明显不同的语言人格
3. 符合2013年6月30日的时代语境
4. 真实事件的基本事实准确
5. 数量达到要求

不能为了事实安全，把所有标题写成冷冰冰的通讯社陈述句。

## 一、先研究“真实标题”，而不只是研究事件

生成数据前必须联网检索。

事件考据来源包括：

- 中文维基百科“2013年”及相关事件页面
- 仍然在线的2013年新闻文章和专题
- 新华网、人民网等新闻来源
- 其他可靠历史资料

但研究标题语气时，必须重点查看：

- Wayback Machine中2013年6月前后的新浪、网易、腾讯、搜狐首页和频道页
- 2013年的科技媒体首页、频道页和专题页
- 2013年的17173、游民星空、游侠网、电玩巴士等游戏媒体页面
- 2012至2013年的贴吧、论坛、地方社区和早期知乎页面
- 2013年的教育、高考、毕业专题页面
- 当时真实存在的广告、导航、排行榜和专题标题

不能只搜索“2013年发生了什么”。

至少收集并分析100条真实的2013年标题样本，涵盖：

- 严肃新闻
- 门户焦点
- 评论和解读
- 科技数码
- 游戏媒体
- 教育专题
- 论坛帖子
- 早期问答社区
- 广告文案

不需要大段复制这些标题。应当把它们归纳为“标题结构”和“编辑方法”。

第一轮只输出：

1. 查阅的网站类型和大致时间范围；
2. 2013年各类网站的标题结构矩阵；
3. 常见标题开头、标点、长度和悬念方式；
4. 各栏目应当采用的不同语言人格；
5. 可能使用的事件和话题池；
6. 已经流行和尚未流行的网络词；
7. 你准备怎样防止同质化。

第一轮不得生成正式Mock Data。

## 二、先进行风格试写，禁止直接批量生产

我回复“开始风格试写”后，只生成以下试样：

- world：16条
- tech：16条
- games：16条
- forum：20条
- gaokao：12条

这一步只用于确认风格，不生成完整文件。

试写结束后，你必须自行检查：

- 是否连续出现相同句式
- 是否过多使用“某国/某人 + 动作 + 结果”
- 是否像年表而不是门户首页
- 是否所有标题都过于客观
- 是否论坛标题仍然像新闻编辑写的
- 是否大量使用同一种网络词
- 是否存在现代互联网语气

我确认风格后，才允许正式生成完整数据。

## 三、必须区分“事件”与“标题角度”

一个真实事件不是只能生成一条电讯式标题。

重大事件最多可以生成3条候选标题，但三条必须采用不同的编辑角度，例如：

- 事实进展
- 原因或背景分析
- 影响与后果
- 当事人或普通人的处境
- 各方反应
- 争议和疑问
- 现场、图集或时间线
- 面向中国读者的解释

错误示范：

- “埃及总统执政周年前夕多地发生冲突”
- “斯诺登离开香港飞抵莫斯科”
- “南非民众为病危中的曼德拉祈福”

这只是同一套电讯句式的重复。

正确的多角度思路类似：

- “埃及政局再次走到十字路口”
- “厄瓜多尔把难题推回莫斯科，斯诺登去向再陷僵局”
- “病房外的鲜花没有停：南非继续等待曼德拉消息”

以上只是展示编辑角度，不得机械模仿这些句子。

标题可以制造悬念、冲突、疑问和阅读动机，但不能歪曲真实事实。

## 四、全局反同质化硬规则

每个文件都必须满足：

1. 连续两条不得采用相同句式。
2. 每10条至少出现6种不同标题结构。
3. “人物或国家 + 动作 + 事件结果”的纯陈述句不得超过全部标题的30%。
4. 每10条至少包括：
    - 2条硬新闻
    - 2条分析或解读
    - 1条疑问或争议
    - 1条影响或后果
    - 1条现场、人物或生活角度
    - 1条专题、回顾、盘点或背景资料
    - 其余可按栏目特点自由分配
5. 每10条至少有：
    - 2条问句
    - 1条带冒号的双段式标题
    - 1条较短标题
    - 1条较长标题
6. 同一个固定短语在同一文件中最多出现3次。
7. “网友热议”“专家称”“引发关注”“仍在继续”“各方反应”等安全套话，每个文件最多出现2次。
8. 不得连续以同一个国家、企业、品牌或人物名称开头。
9. 标题长度必须有明显变化，不能像数据库自动摘要。
10. 不得为了凑数量，把同一事件的时间线逐日拆成多条近义标题。

## 五、增加标题类型字段

正式数据增加headlineType字段。

headlineType只能使用：

"hard-news"
"analysis"
"explainer"
"impact"
"reaction"
"human"
"scene"
"service"
"comparison"
"rumor"
"review"
"guide"
"controversy"
"forum"
"question"
"advert"

示例：

{
"id": "world-001",
"date": "06/30",
"title": "示例标题",
"headlineType": "analysis",
"authenticity": "adapted",
"tone": "portal",
"hot": 2,
"basis": "R-WORLD-001"
}

这个字段即使前端暂时不用，也必须保留，用于检查标题类型是否失衡。

## 六、不同栏目必须有不同编辑人格

### 1. world：国际新闻，40条

world不是国际大事年表。

内容应基于真实事件，但多数标题应为adapted，而不是直接照搬通讯社标题。

建议构成：

- 12条硬新闻
- 8条分析和原因
- 5条背景解释
- 4条影响和后果
- 4条人物、现场或普通人角度
- 3条各方反应
- 2条争议问句
- 2条科学、文化或国际趣闻

允许使用：

- “为何……”
- “意味着什么”
- “局势会否……”
- “背后有哪些原因”
- “一场……正在改变……”
- “从……到……”
- “现场：……”
- “背景资料：……”
- “各方如何回应”
- “谁将成为赢家”

不能让40条全部由国家、领导人、机构名称开头。

### 2. china：中国新闻，45条

不能全部是政策、会议、事故和通报。

建议包含：

- 硬新闻
- 政策对普通人的影响
- 社会争议
- 城市和地方生活
- 服务提醒
- 人物故事
- 经济、就业和房价讨论
- 天气、交通、灾害和公共安全
- 门户专题和评论角度

部分标题可以有民生提问，例如：

- 新规定对普通人有什么影响
- 某种社会现象为什么引发争议
- 某地变化将给居民带来什么

### 3. tech：科技数码，45条

必须像2013年的科技频道，而不是产品发布年表。

建议构成：

- 8条行业新闻
- 7条产品或系统进展
- 7条评测、体验和实测
- 6条品牌或设备对比
- 5条真实存在的传闻
- 5条行业分析
- 4条用户争论
- 3条技巧、盘点或购买建议

应当出现不同结构：

- “传……”
- “……究竟值不值得买”
- “实测：……”
- “五个细节看懂……”
- “安卓阵营为何……”
- “PC真的会被平板取代吗”
- “运营商准备好迎接4G了吗”
- “小米、魅族和传统厂商谁更懂年轻人”

不要把每条都写成“某公司发布某产品”。

### 4. games：游戏资讯，50条

必须具有游戏媒体的编辑人格。

建议混合：

- 正式新闻
- E3和次世代主机观察
- 游戏前瞻和评测
- 攻略和盘点
- 公测、开服和激活码
- 玩家吐槽
- 版本争议
- 主机与PC之争
- 网游、单机、页游、手游之间的争论
- 游戏行业分析
- 编辑部轻松标题

允许少量使用：

“小编”
“玩家吐槽”
“坑爹”
“给力”
“节操”
“完爆”
“有图有真相”
“你怎么看”

### 5. gaokao：高考与志愿填报，35条

不能全部是“某地公布分数线”。

需要同时包含：

- 查分和分数线

- 平行志愿和录取规则
- 专业选择
- 家长与考生冲突

## 七、真实性和时间边界

网站站在2013年06月30日当天。

**禁止**：

- 使用07/01及之后才发生的事实
- 提前知道2013年下半年的结果
- 使用后来才发布的产品和版本
- 使用现代视角评价后来结果
- 使用2020年代网络语言
- 只生成2013-06这一个月内的

日期格式：

"06/30"
"05/04"
"02/12"

date表示这条标题在网站上的发布日期。

旧事件可以在6月底以以下方式重新出现：

- 事件追踪
- 背景资料
- 半年回顾
- 最新进展
- 影响分析
- 专题盘点

总体日期分布建议：

- 40%：06/24至06/30
- 25%：06/01至06/23
- 20%：04/01至05/31
- 15%：01/01至03/31

高考、毕业、夏季、论坛模块应更多集中在6月下旬。

较早的新闻可以进入数据池，但若被选入焦点区，应自然写成“事件追踪”“专题”“回顾”“最新进展”等形式，不能让数月前的普通新闻伪装成刚发生的突发新闻。

真实性分为：

"real"
基于真实事件的直接新闻标题。

"adapted"
基于真实事件，但采用分析、疑问、影响、人物、现场或专题角度重新编辑。

"fictional"
符合时代环境的虚构论坛、生活资讯、问题和广告。

world、china、tech和games中，adapted应当多于real。

不得生成正文、摘要、虚假链接或图片说明。

## 十、正式生成方法

不要一次输出三个包含数十条数据的文件。

每次只生成一个模块，每批结束报告：

- 各headlineType数量
- real、adapted、fictional数量
- 使用了多少个不同事件
- 有多少问句、分析标题、人物标题和纯陈述句
- 出现频率最高的5个词
- 是否有连续相似句式
- 是否需要主动重写部分条目

## 十一、最终目标

生成结果应当像：

一个2013年6月30日仍在运行的中文综合门户网站首页候选池。

首页中既有严肃新闻，也有解读、悬念、人物、专题、争议、游戏媒体标题、实用信息、普通人的琐碎发言和商业噪声。

它不能像：

- 维基百科事件列表
- 通讯社电讯目录
- 现代信息流营销号
- 由同一个模板批量改名生成的标题数据库

---

# fe_dev.js

你现在扮演“2013年中文开发者媒体编辑、Web前端工程师和互联网技术考据员”。

我要为一个虚构的2013年中文综合门户网站制作一个独立的“开发者频道”标题数据池。

网站当前日期固定为：

2013年06月30日

这个频道面向2013年的中国程序员和网站开发者，内容以前端开发为主，同时适度涉及浏览器、JavaScript生态、Node.js、开源工具、网站性能、服务器端技术和程序员社区讨论。

网站用户主要使用：

- IE8、IE9、IE10
- 360安全浏览器的兼容模式或极速模式
- Chrome
- Firefox
- 搜狗浏览器、QQ浏览器等国产浏览器

网站自身采用ES5、jQuery 1.x、Bootstrap 2、table和ul布局，但频道内容可以报道当时正在出现的新技术。

你的任务不是写教程正文，也不是罗列软件版本号，而是生成：

“2013年6月30日当天，一个中文开发者媒体首页可能出现的新闻、教程、讨论和观点标题候选池”。

最终只输出标题数据，不输出正文。

- Microsoft IE Blog、MSDN
- Node.js官方博客

- 相关语言、框架或工具的官方文档和历史版本记录

研究中文标题语气时，重点查看2012年至2013年的：

- CSDN
- InfoQ中文站

2. 当时是否已经发布正式版
3. 当时只是预览版、Alpha、Beta、传闻还是提案
4. 当时中文开发者是否已经广泛讨论
5. 现代开发者对它的称呼是否和2013年一致

不能因为某项技术后来非常重要，就让2013年的编辑提前知道它的历史地位。

## 六、反同质化规则

必须严格执行：

1. 连续两条不能采用同一种句式
2. 每10条至少出现6种不同标题结构
3. 纯粹的“项目名 + 发布版本 + 新功能”标题不得超过总数的25%
4. 问句不得超过总数的35%
5. 冒号标题不得超过总数的35%
6. 不能让所有标题都使用“能否”“为何”“到底”
7. 同一个技术最多生成3条，且必须是不同角度
8. 同一个事件不能只改几个词重复生成
9. 标题长度必须有明显变化，约15至34个汉字为主，允许少量更短或更长
10. 至少有8条标题面向普通网站开发者的实际工作，而不是只报道框架和版本
11. 至少有6条涉及IE兼容、浏览器差异或旧网站维护
12. 至少有6条涉及开发者职业、社区或工作方式
13. 不得把全部内容写得像科技公司新闻稿
14. 不得把全部内容写成疑问句或观点文章

## 七、日期规则

日期格式使用：

"06/30"
"06/25"
"04/18"

日期表示该标题在虚构网站上的发布日期。

日期分布建议：

- 30%：06/01至06/30
- 30%：04/01至05/31
- 20%：01/01至03/31
- 20%：作为06月底重新发布的教程、回顾或讨论

较早事件若日期写为06月底，标题必须体现：

- 回顾
- 观察
- 使用体验
- 几个月后的开发者反馈
- 兼容性总结
- 是否值得升级
